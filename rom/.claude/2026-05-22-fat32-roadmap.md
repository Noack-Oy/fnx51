# FAT32: status and proposed next steps

## Context

The fatfs/ layer currently mounts a FAT32 volume and converts cluster
numbers to LBAs — nothing else. Everything that an actual filesystem
user (open, read, enumerate, etc.) needs is unimplemented. The
infrastructure below it (SD driver, single-block cache, heap, 32-bit
arith, dual-dptr utils) is in place and exercised by tests.

What exists:
- `rom/fatfs/info.equ` — layout of the 21-byte `fatfs_info` struct
  (partition LBA, sectors-per-cluster, FAT1/FAT2 LBA, cluster-0 LBA
  pre-adjusted, root-dir cluster number).
- `rom/fatfs/init.inc` — `fatfs_init` parses the MBR, validates
  partition type 0x0b/0x0c, reads the FAT32 boot sector, validates
  bytes-per-sector=512, sectors-per-cluster=power-of-two, num-FATs=2,
  and populates `fatfs_info`.
- `rom/fatfs/cluster.inc` — `fatfs_cluster_to_lba` returns the first
  LBA of a given cluster.
- `rom/test/fatfs.t.py` only confirms the struct dump matches the
  reference SD card and that cluster 2 → LBA 0x4080.

What is missing (everything above mount):
- Reading the FAT (next-cluster lookup, EOF/free detection).
- Directory entry parsing (32-byte records, end-of-dir sentinel,
  deleted-entry marker, attribute byte, 8.3 names, start-cluster
  hi/lo split, file size).
- File handle concept and read API.
- Path resolution (probably just root-dir-only first).
- Writes (cluster allocation, dir-entry creation, append, delete).
- LFN parsing.

The user has chosen the read-first roadmap with writes as a clearly
separated later phase.

## Proposed next steps (read-first)

Three PR-sized steps, each independently testable, in dependency
order. They're listed with the registers / signatures I'd expect to
write — open to revision when we actually do them.

### Step 1 — `fatfs/chain.inc`

Introduce a "chain handle" abstraction that hides cluster-chain
traversal from everything above it. Directories and files both
look like a linear sequence of 512-byte blocks; the handle owns the
state needed to map a relative block number to an LBA. In step 1 we
ship the API and plumbing, but stub the actual traversal — it lands
the first time a caller needs it.

#### Lifecycle convention (used by chain, dir, file)

Steps 2 and 3 want their handles to be "extensions" of the chain
handle — the chain header sits at offset 0, with extra fields tacked
on the end. To make that work cleanly without reallocation, we split
each handle type's lifecycle into two layers:

- `*_init(self{dptr}, ...args)` — initialize a pre-allocated struct.
  No allocation. Used by extension types that own a larger
  allocation.
- `*_deinit(self{dptr})` — release any sub-resources held by the
  struct. No deallocation. Pairs with `_init`. Currently a no-op for
  the chain handle since it owns nothing besides its own bytes, but
  declared so extension types have a consistent "destructor to call
  before freeing the outer allocation".
- `*_open(...args) → self{dptr}` — convenience for standalone use:
  `memory_allocate` + `_init`.
- `*_close(self{dptr})` — convenience for standalone use: `_deinit`
  + `memory_release`.

Extension structs (`dir`, `file`) only call the chain's `_init` /
`_deinit`; they never call `chain_open`/`chain_close`, because that
would double-allocate or double-free.

#### Chain handle

Handle layout (8 bytes):
- bytes 0..3: first cluster number (uint32) — never changes after
  init; lets us cheaply re-anchor.
- bytes 4..7: current cluster number (uint32) — last cluster we
  resolved to. Updated by block-resolution.

We may add a third field (current cluster's index within the chain)
when we wire up the real traversal — without it we can't tell
whether the requested block lies before or after `current`. Settle
when we get there; not needed for the stub.

`fatfs_chain_init(handle{dptr}, first_cluster{r0-r3})`
- Write `first_cluster` into both the first- and current-cluster
  fields of the pre-allocated 8-byte region at `handle`.

`fatfs_chain_deinit(handle{dptr})`
- No-op for now. Stub present so extension types can call it
  unconditionally.

`fatfs_chain_open(uint32{r0-r3} first_cluster) → handle{dptr}`
- `memory_allocate(8)`, `fatfs_chain_init`, return the pointer.

`fatfs_chain_close(handle{dptr})`
- `fatfs_chain_deinit`, then `memory_release`.

#### Block access

We expose two functions. The primary primitive is `ensure_block`,
which loads a block into the SD/block cache and hands back a
borrowed dptr into the cache — the caller copies out whatever it
needs and must not retain the pointer across other `block_load`
calls. `load_block` is a thin convenience that adds a 512-byte copy
on top.

`fatfs_chain_ensure_block(handle{dptr}, block{r0-r3}) → cached{dptr}`
- Intended semantics (when fully implemented):
  - Compute `cluster_idx = block / fatfs_info.sectors_per_cluster`
    and `sector_in_cluster = block % sectors_per_cluster`.
  - If `cluster_idx == 0`, the target cluster is the first cluster
    (known directly from the handle).
  - Else if `cluster_idx` matches the current cluster's chain index,
    the target cluster is the current cluster.
  - Else if `cluster_idx` is ahead of current, walk forward from
    current via `fatfs_chain_next_cluster`, updating the current-
    cluster field as we go.
  - Else (cluster_idx is before current), restart from the first
    cluster and walk forward.
  - Once we've resolved the target cluster, `fatfs_cluster_to_lba`
    + `sector_in_cluster` gives the LBA. `block_load` it; return
    the dptr it gave us (a borrow into the block cache).
- Step 1 stub: if `block == 0`, look up the LBA of the first block
  of the first cluster via `fatfs_cluster_to_lba`, `block_load` it,
  return that dptr. Otherwise `panic`. This is enough for steps 2/3
  to start using the API; the real traversal (and
  `fatfs_chain_next_cluster`) will be filled in when one of them
  first reads past the first cluster.

`fatfs_chain_load_block(handle{dptr}, block{r0-r3}, dst{dptr'})`
- Convenience: `fatfs_chain_ensure_block`, then memcpy 512 bytes
  from the cached sector to `dst`. May or may not see callers in
  practice — dir iter and file read both prefer `ensure_block` so
  they can copy just the bytes they actually need.

`fatfs_chain_next_cluster` (FAT-entry lookup) is intentionally not
built in step 1. When `fatfs_chain_ensure_block`'s body is fleshed
out, this routine joins it in `rom/fatfs/chain.inc` (the
`fatfs_chain_` prefix keeps the file it lives in obvious from the
label):

`uint32{r0-r3} fatfs_chain_next_cluster(uint32{r0-r3} cluster)`
- Compute FAT-entry sector LBA: `fatfs_info.fat1_lba + (cluster /
  128)` (128 = 512/4 four-byte entries per sector).
- `block_load` that LBA, advance dptr by `(cluster % 128) * 4`, read
  four bytes, mask off the high nibble (FAT32 reserved bits).
- Return the value; any value ≥ `0x0ffffff8` means end-of-chain.

Test idea (`test/fatfs.t.py` extension): open a chain on a known
start cluster from the reference card (e.g. the root dir cluster),
`ensure_block(0)`, copy the first 32 bytes (one directory entry)
out of the borrowed pointer, dump the name field. Exercises
open/init/ensure_block/close end-to-end without depending on chain
traversal.

### Step 2 — `fatfs/dir_iter.inc`

Walks 32-byte directory entries in a cluster chain, returning one
entry at a time. The dir handle is an *extension* of the chain
handle from step 1: the chain header sits at offset 0, dir-iter
fields follow, and the whole thing is one allocation owned by the
dir handle (no realloc, no separate sub-allocations).

Dir handle layout (single allocation, ~45 bytes):
- bytes 0..7: chain handle (first cluster, current cluster) —
  initialized via `fatfs_chain_init`.
- bytes 8..11: relative block number within the chain (uint32).
- byte 12: entry index within the current 512-byte block (0..15).
- bytes 13..44: embedded 32-byte entry buffer that `dir_next`
  returns to callers.

The embedded buffer lets `dir_next` return a pointer the caller
*owns* the lifetime of (valid until the next `dir_next` or
`dir_close` on the same handle). We can't hand out a pointer
straight into the block cache because we don't own its eviction
policy.

`fatfs_dir_open(uint32{r0-r3} start_cluster) → handle{dptr}`
- `memory_allocate` the full extended struct.
- `fatfs_chain_init` on bytes 0..7 with `start_cluster`.
- Set block=0, entry=0. (The embedded buffer is left undefined;
  it'll be populated by the first `dir_next`.)
- Return the pointer.

`fatfs_dir_next(handle{dptr}) → entry{dptr'}, status{a}`
- If we've consumed all 16 entries in the current block, advance
  the block counter and reset entry index.
- Call `fatfs_chain_ensure_block` to make sure the current block is
  in the cache; that returns a borrowed dptr into the cache.
- Memcpy 32 bytes from `(cached + entry_index*32)` into the dir
  handle's embedded entry buffer. Advance `entry_index`.
- Inspect the copied entry:
  - first byte 0x00 → status=1 (end of directory). Don't advance.
  - first byte 0xE5 → deleted, loop and try the next entry.
  - attribute byte 0x0f → LFN, skip for now (log if useful), loop.
  - otherwise → status=0, return dptr' pointing at the embedded
    buffer.
- Status=2 path (chain exhausted without sentinel) is a corner
  case; specifics depend on how `ensure_block` signals "no such
  block". Settle when we implement traversal.

`fatfs_dir_close(handle{dptr})`
- `fatfs_chain_deinit` on the chain prefix (currently a no-op), then
  `memory_release` the whole struct. No separate chain free —
  there's only one allocation.

Test idea: walk the root directory on the reference card and dump
every short-name entry's 8.3 name, attribute, size, start cluster.

### Step 3 — `fatfs/file_open.inc` + `fatfs/file_read.inc`

Combines the above into the actual file I/O the user cares about.

The file handle follows the same extension pattern as the dir
handle — chain header at offset 0, file-specific fields after, one
allocation.

File handle layout (single allocation, ~16 bytes):
- bytes 0..7: chain handle.
- bytes 8..11: total file size (uint32, from the dir entry).
- bytes 12..15: current byte offset within the file (uint32).

`fatfs_file_open(name{dptr}) → handle{dptr}, status{a}`
- Walk the root directory via the iterator, compare each entry's 8.3
  name against `name` (11-byte padded form). Eventually generalize
  to path resolution; for the first cut, root only.
- On match, `memory_allocate` the file struct, `fatfs_chain_init` on
  the prefix with the entry's start cluster, copy total size from
  the dir entry, zero the offset. Return the pointer.
- On no-match or end-of-dir, return status≠0.

`fatfs_file_read(handle{dptr}, n{r0r1}, dst{dptr'})`
- Tricky because we have only two dptr roles per call. Likely
  signature: handle in `dptr`, byte count in `r0r1`, dest in dptr'
  (`inc auxr1; mov dpl/dph; inc auxr1`). Details settle in step-3
  design.
- Reads at most `n` bytes (returning actual count in `r0r1`). For
  each iteration: derive (block_in_chain, offset_in_block) from the
  current byte offset, call `fatfs_chain_ensure_block` on the
  embedded chain header, copy the relevant bytes out of the cache
  to `dst`. Cluster crossings are handled inside `ensure_block`.

`fatfs_file_close(handle{dptr})`
- `fatfs_chain_deinit` on the chain prefix, then `memory_release`
  the whole struct.

Test idea: `cat` a known small text file from the reference card
through the test program over the serial port.

## Future phase — writes (not in this round)

When we revisit:
- `fatfs_alloc_cluster` — scan the FAT for the first 0x00000000
  entry, mark it as end-of-chain, return its number. Naive linear
  scan; optimize later with a hint cursor.
- `fatfs_free_chain(cluster)` — walk the chain, zero each entry.
- `fatfs_dir_create_entry` — find a free slot in a directory (`0xE5`
  or `0x00`), write a new entry.
- `fatfs_file_write` / `fatfs_file_append` — extend the chain,
  update size in the directory entry.
- FAT mirroring: writes need to update both FAT1 and FAT2.
- `block_flush` / `block_dirty` (`rom/block/flush.inc` and
  `dirty.inc` are stubs that just panic) — these need real
  implementations before any write code can be tested.

LFN can stay deferred; short names suffice for the loader use case.

## Verification

Each step adds expectations to `rom/test/fatfs.t.py`, which already
runs against the real Raspberry Pi harness and a known reference SD
card. Run `make test` in `rom/test/` — the dependency tracking in
the Makefile only rebuilds and reruns what changed, so iterating on
a single step is cheap. The test card's layout is fixed and known,
so hard-coded expected values work fine.

For the read path, the highest-confidence end-to-end test is
"`cat`-ing" a known file in step 3: type the filename over serial,
read the contents back, diff against the file on the SD card.
