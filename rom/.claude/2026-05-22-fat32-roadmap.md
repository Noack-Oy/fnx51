# FAT32: status and proposed next steps

## Context

The fatfs/ layer currently mounts a FAT32 volume and converts cluster
numbers to LBAs — nothing else. Everything that an actual filesystem
user (open, read, enumerate, etc.) needs is unimplemented. The
infrastructure below it (SD driver, single-block cache, heap, 32-bit
arith, dual-dptr utils) is in place and exercised by tests.

What exists:
- `rom/fatfs/info.equ` — layout of the 21-byte `fatfs_info` struct
  (partition LBA, sectors-per-cluster, FAT1/FAT2 LBA, cluster-0 LBA
  pre-adjusted, root-dir cluster number).
- `rom/fatfs/init.inc` — `fatfs_init` parses the MBR, validates
  partition type 0x0b/0x0c, reads the FAT32 boot sector, validates
  bytes-per-sector=512, sectors-per-cluster=power-of-two, num-FATs=2,
  and populates `fatfs_info`.
- `rom/fatfs/cluster.inc` — `fatfs_cluster_to_lba` returns the first
  LBA of a given cluster.
- `rom/test/fatfs.t.py` only confirms the struct dump matches the
  reference SD card and that cluster 2 → LBA 0x4080.

What is missing (everything above mount):
- Reading the FAT (next-cluster lookup, EOF/free detection).
- Directory entry parsing (32-byte records, end-of-dir sentinel,
  deleted-entry marker, attribute byte, 8.3 names, start-cluster
  hi/lo split, file size).
- File handle concept and read API.
- Path resolution (probably just root-dir-only first).
- Writes (cluster allocation, dir-entry creation, append, delete).
- LFN parsing.

The user has chosen the read-first roadmap with writes as a clearly
separated later phase.

## Proposed next steps (read-first)

Three PR-sized steps, each independently testable, in dependency
order. They're listed with the registers / signatures I'd expect to
write — open to revision when we actually do them.

### Step 1 — `fatfs/next_cluster.inc`

The foundation for any traversal. Without this, neither directories
nor files can span more than one cluster.

`uint32{r0-r3} fatfs_next_cluster(uint32{r0-r3} cluster)`

- Compute FAT-entry sector LBA: `fatfs_info.fat1_lba + (cluster /
  128)` (128 = 512/4 four-byte entries per sector).
- Call `block_load` with that LBA. `block_load` returns the cached
  sector in dptr; reuse the cache transparently.
- `dptr += (cluster % 128) * 4`; read four bytes as the next-cluster
  number. Mask high nibble (FAT32 reserves it).
- Return value, with `0x0fffffff` (or any value ≥ 0x0ffffff8) meaning
  end-of-chain. Caller checks before re-entering.

Test idea (`test/fatfs.t.py` extension): given the known reference
card, walk a few clusters of a known file and check the chain. Or
just probe FAT[2], FAT[3] for known values from a hexdump of the
volume.

### Step 2 — `fatfs/dir_iter.inc`

Walks 32-byte directory entries in a cluster chain. Returns one
entry at a time via dptr.

State: a small struct (4-6 bytes) in xram tracking current cluster,
current sector within cluster, current entry-offset within sector.

`fatfs_dir_open(uint32{r0-r3} start_cluster) → handle{dptr}`
- Allocates a small dir-iterator state on the heap, initializes it
  to the first sector of the first cluster, returns a handle pointer.

`fatfs_dir_next(handle{dptr}) → entry{dptr}, status{a}`
- Loads the current sector via `block_load`, points dptr at the
  current 32-byte entry, advances internal state.
- Skips deleted entries (first byte = 0xE5).
- Skips LFN entries (attribute byte 0x0f) for now — log them but
  return only short entries.
- Status code: 0 = entry valid, 1 = end of directory (first byte =
  0x00), 2 = no more sectors (chain ended without sentinel).

`fatfs_dir_close(handle{dptr})`
- `memory_release` the iterator state.

Test idea: walk the root directory on the reference card and dump
every short-name entry's 8.3 name, attribute, size, start cluster.

### Step 3 — `fatfs/file_open.inc` + `fatfs/file_read.inc`

Combines the above into the actual file I/O the user cares about.

`fatfs_file_open(name{dptr}) → handle{dptr}, status{a}`
- Walk the root directory via the iterator, compare each entry's 8.3
  name against `name` (11-byte padded form). Eventually generalize
  to path resolution; for the first cut, root only.
- On match, allocate a file-handle struct (start cluster, total
  size, current cluster, current sector within cluster, current byte
  offset within sector — ~12 bytes). Return pointer in dptr.
- On no-match or end-of-dir, return status≠0.

`fatfs_file_read(handle{dptr}, n{r0r1}, dst{dptr-on-second-arg?})`
- Tricky because we have only two dptr roles per call. Likely
  signature: handle in `dptr`, byte count in `r0r1`, dest in the
  alternate dptr (`inc auxr1; mov dpl/dph; inc auxr1`).
  Will work out details in step-3 design.
- Reads at most `n` bytes (returning actual count in `r0r1`).
  Follows the cluster chain via `fatfs_next_cluster` when crossing
  cluster boundaries.

`fatfs_file_close(handle{dptr})`
- `memory_release` the handle.

Test idea: `cat` a known small text file from the reference card
through the test program over the serial port.

## Future phase — writes (not in this round)

When we revisit:
- `fatfs_alloc_cluster` — scan the FAT for the first 0x00000000
  entry, mark it as end-of-chain, return its number. Naive linear
  scan; optimize later with a hint cursor.
- `fatfs_free_chain(cluster)` — walk the chain, zero each entry.
- `fatfs_dir_create_entry` — find a free slot in a directory (`0xE5`
  or `0x00`), write a new entry.
- `fatfs_file_write` / `fatfs_file_append` — extend the chain,
  update size in the directory entry.
- FAT mirroring: writes need to update both FAT1 and FAT2.
- `block_flush` / `block_dirty` (`rom/block/flush.inc` and
  `dirty.inc` are stubs that just panic) — these need real
  implementations before any write code can be tested.

LFN can stay deferred; short names suffice for the loader use case.

## Verification

Each step adds expectations to `rom/test/fatfs.t.py`, which already
runs against the real Raspberry Pi harness and a known reference SD
card. Run `make test` in `rom/test/` — the dependency tracking in
the Makefile only rebuilds and reruns what changed, so iterating on
a single step is cheap. The test card's layout is fixed and known,
so hard-coded expected values work fine.

For the read path, the highest-confidence end-to-end test is
"`cat`-ing" a known file in step 3: type the filename over serial,
read the contents back, diff against the file on the SD card.
